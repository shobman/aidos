# Migration v1.2.0 → v1.3.0

## Summary

Altitude refactor + subtraction. Existing artifacts keep working. The framework's discipline tightens around what belongs in Tech Design (architecture, not implementation) and Testing (behaviour, not test code), and two artifacts that never landed in real use retire.

**Headline changes:**

1. **New Core C13 — Implementation neutrality at the right altitude.** Single cross-cutting rule that subsumes the old P11 and S10 implementation-neutrality criteria.
2. **P11 (Problem) retires** as "Implementation neutrality" — concern moves to C13.
3. **P11 slot is reused** for the new **Honest framing** criterion. Old P11 ≠ new P11. The migration leaves this slot empty in existing files unless explicitly upgraded.
4. **S10 (Solution) retires** as "Implementation neutrality" — concern moves to C13. S10 slot is left vacant in v1.3.0.
5. **Tech Design rubric A1–A10 reframed** at architectural altitude. A5 (Technology choices) and the old A8 (Migration path bookkeeping) retire. Four load-bearing criteria sharpen: A2 Seam contracts (kind, not shape), A3 State ownership and topology, A4 Invariants (NEW), A9 Implementation handoff. New criteria: A4 Invariants, A7 Temporal stance, A8 Trust zones, A10 Transition strategy.
6. **Testing rubric T1–T9 reframed** at behavioural altitude. Test-code mechanics (specific HTTP codes, payload shapes, framework usage, file paths) explicitly belong in the coding session. T1–T9 reframe as behavioural coverage / traceability / scenarios as Given-When-Then / expected behaviour / preconditions as state / where assertions hold / behavioural regression scope / risk-based prioritisation.
7. **Conditional sections.** Root Cause Analysis, Scope, Existing Alternatives (Problem); Alternatives Considered, Constraint Compliance, Migration and Transition (Solution); new Scope Boundary section in Solution. Absence of a conditional section is not a Fail; unearned presence is a Bug.
8. **Definition artifact retires entirely.** Delivery artifacts stay current as the long-term record. Existing Definition files in `.aidos/definitions/` or `.aidos/archive/` folders: leave them — they don't break anything; future projects just don't create them.
9. **Retrospective template retires.** Retros still happen conversationally; just not from a template.

**Backward compatibility.** Existing artifacts that passed v1.2.x rubrics will still pass v1.3.0 in some sensible way — the rubrics tighten but don't break. Re-audit at your discretion; not required.

## File renames

None.

## Content changes

For Problem artifacts (`problem.md` or combined documents containing a Problem section):
- The "## Root Cause Analysis", "## Scope", and "## Existing Alternatives" sections are now CONDITIONAL — keep them if their content is real and earned; remove them if they were stub-only.
- The P11 criterion's old meaning ("Implementation neutrality") no longer applies — its concern is captured by Core C13 across the whole artifact. New P11 (Honest framing) requires the Problem to read honestly about what's broken, including awkward truths the author would have reason to soften. Existing Problem artifacts may pass the new P11 if their prose already reads honestly; if it reads as sanitised vendor-pitch prose, surface the messier version.

For Solution artifacts (`solution.md` or combined documents containing a Solution section):
- The "## Alternatives Considered", "## Constraint Compliance", and "## Migration and Transition" sections are now CONDITIONAL.
- Consider adding the new "## Scope Boundary" section ONLY if adjacent responsibilities could be confused with this one.
- S10's old criterion ("Implementation neutrality") no longer applies — captured by C13.

For Tech Design artifacts (`tech-design.md` or combined documents containing a Tech Design section):
- Section structure changes: "## Components" → "## Boundaries"; "## Integration Points" → "## Seam Contracts"; "## Data Model" → "## State Ownership"; "## Error Handling" → "## Failure and Recovery"; "## Technology Choices" → DELETE; "## Performance and Capacity" → "## Quality Properties"; "## Deployment and Environment" → DELETE; "## Migration Path" → "## Transition Strategy"; "## Constraints and Boundaries" → fold into "## Implementation Handoff"; "## Coding Agent Brief" → "## Implementation Handoff".
- New sections to consider: "## Invariants", "## Temporal Stance", "## Trust Zones".
- Existing content can usually map to a renamed section; what's hardest to preserve is altitude — content that was implementation-grounded (specific schemas, library choices, deployment infra) belongs in the coding session, not the Tech Design.

For Testing artifacts (`testing.md` or combined documents containing a Testing section):
- Section structure changes: "## Test Scenarios" → "## Behavioural Assertions"; "## Test Data and Preconditions" → "## Required State"; "## Environment Requirements" → "## Where Assertions Hold". The other sections keep their names but altitude tightens.
- Reframe test-code-shaped assertions ("API returns 404 with body {...}") to behavioural ones ("an unauthenticated request is refused with an authorisation error").

For Definition artifacts (`definition.md`):
- The artifact type retires entirely. Existing Definition files remain valid markdown — they don't break anything. The framework no longer creates new ones or expects them at project closure.

For Retrospective files (`retrospective.md`):
- The template retires entirely. Existing retrospective files remain valid markdown.

## Metadata changes

Update the `**AIDOS Version:**` line in every artifact's metadata block from `1.2.0` (or `1.2.1`) to `1.3.0`. This is mechanical — the per-file upgrade flow in the builder handles it.
